## Problem Statement
The current manual testing, building, and releasing process is slow and inconsistent. Users also have no way to know when a new version is available, leading to fragmentation and missed security patches.

## Demand Evidence
"this system for share a video on youtube and and whatch that on browser" - The core use case is clear, and the developer is actively iterating to improve the setup UX.

## Status Quo
Manual `node index.js`, manual `pkg` builds, manual `gh release create`. Users must manually check GitHub for updates.

## Target User & Narrowest Wedge
Individual users who want to share links from mobile to PC browser. The narrowest wedge is a single binary that handles everything from setup to updates.

## Constraints
- Must support Windows and Mac.
- Must use GitHub Actions for automation.
- Must remain lightweight.

## Premises
1. The primary pain point is the friction of manual copy-pasting links between mobile and PC.
2. Discord is a better transport than OS-level tools because it's account-neutral and cross-platform.
3. Users want a "surgical" opener that doesn't require syncing their entire browser history.

## Implementation Details
- **Source of Truth**: Current version is strictly read from `package.json`.
- **Builder**: Use `@yao-pkg/pkg` for Node 22 support in GitHub Actions.
- **Asset Naming**: Assets follow `DiscordLinkOpener-{OS}-{ARCH}-{VERSION}.exe/bin` for deterministic mapping.
- **Integrity**: CD pipeline generates SHA-256 checksums; app verifies integrity before suggesting execution.
- **API Resilience**: Update checks use ETag conditional requests or a local 24-hour cache to respect GitHub API rate limits.
- **Platform Specifics**:
    - **Windows**: Use a detached `.bat` script to replace the locked `.exe` file.
    - **Mac**: Provide "Right-click -> Open" instructions to bypass Gatekeeper.
- **Rollback & Recovery**: Update scripts must create a `.bak` copy of the current binary before replacement. If the new binary fails to start (checked via a 10s health check), the script automatically restores the `.bak` file.
- **Suggesting Execution**: The app notifies the user in the Web UI: "A new version is available. [Download & Update]". Execution only happens after explicit user confirmation in the UI.

## Recommended Approach
**Approach C**: Build a complete lifecycle for the bot.
- **CI**: Auto-test on push/PR.
- **CD**: Auto-build and release on version tag.
- **App**: Add an "Update" module (`src/updater.js`) to `index.js` that compares `package.json` version with GitHub Release API and notifies the user in the Web UI.

## Open Questions
- Should the app auto-download the update or just link to it? (Recommendation: Start with link-to-download, evolve to auto-patcher).
- How to handle OS-specific binary selection in the update check? (Solved: Strict naming convention).
